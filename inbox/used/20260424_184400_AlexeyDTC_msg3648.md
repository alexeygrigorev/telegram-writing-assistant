Problem:
When away from computer, creating GitHub issues manually on mobile is slow (choose repo, template, write details).

Solution implemented:
Built a Telegram-based AI agent that accepts voice notes or text, transcribes and sanitizes input, classifies the target repository, drafts a structured GitHub issue, and creates it only after human approval.

How it works (implemented flow):

Configured repository scope via GITHUB_REPOS (curated list of repos).

User sends voice or text message to Telegram bot.

Voice is transcribed using Whisper (whisper-1).

PII is scrubbed using Microsoft Presidio before agent processing.

Agent classifies repo and drafts issue (title/body/labels/confidence).

User sees draft and confirms via Approve & Create / Cancel.

On approval, issue is created via GitHub API; all runs are logged.

Key capabilities delivered:

Voice + text input via Telegram

Repo routing using RAG-style repo search (search_repos tool)

Structured output with schema validation (Pydantic models)

Human-in-the-loop confirmation before write actions

Observability stack: PostgreSQL logs, Streamlit dashboard, Grafana metrics

Async “LLM-as-a-Judge” pipeline scoring interaction quality

Tech stack:

Python, pydantic-ai, OpenAI API (Whisper + LLM), python-telegram-bot

GitHub API (PyGithub)

Presidio (PII detection/anonymization)

Docker Compose microservices (bot, judge, postgres, dashboard, orchestrator, grafana)

Overview

I have build SnapSplit, an AI-powered receipt splitting app.

Upload a photo of a restaurant bill, a Vision model extracts the line items, each person claims what they ordered, and the app calculates who owes what (including shared service charge).

Stack:

FastAPI + PostgreSQL + OpenAI gpt-4.1-mini + Pydantic-AI agent + Marimo interactive UI

How it works internally:

The core of the app is a Pydantic-AI agent (receipt-reconciliation) that orchestrates extraction and verification through 4 tools:

extract_items — Sends the receipt image to one of OpenAI's Vision models (gpt-4.1-mini) using structured outputs (ExtractionResult). The model returns typed line items (name, price, quantity) plus subtotal, tax, service charge, and total.
check_totals_ _— Verifies receipt arithmetic. It tries two interpretations: tax-additive (items + service charge + tax = total) and tax-inclusive (items + service charge = total, tax is informational). If neither matches within a £0.01 tolerance, it builds a diagnostic hint that includes the items and prices from the first extraction, the computed vs expected total, and the size of the discrepancy. If the discrepancy exactly matches the price of a specific item, the hint highlights that item — suggesting it may be spurious (if the total is too high) or missing a second instance (if the total is too low). This hint is stored on the agent's dependency context and automatically injected into the re_extract_items prompt, so the vision model knows exactly where the first pass went wrong and what to look more carefully at.
re_extract_items — Called when check_totals finds a mismatch. It re-runs OCR with the diagnostic hint injected into the prompt (e.g. "previous extraction totalled £45.20 but receipt says £43.20, discrepancy of £2.00 — check for 0/8 digit confusion on thermal receipts"). This uses a more powerful model (gpt-5.2) and higher image detail for the second pass. Limited to 1 re-extraction to prevent loops.
flag_for_human — Escalates to human review when extraction fails, no items are found, or totals still don't match after re-extraction.
The agent follows a strict protocol: extract → verify → (if mismatch) re-extract with hint → verify again → (if still wrong) flag for human. Guard rails prevent infinite loops: check_totals is capped at 2 calls and re_extract_items at 1.

Evals:

- OCR eval (precision/recall/F1) against 9 annotated receipts

- LLM-as-a-Judge evaluation across 9 criteria (completeness, price accuracy, agent behaviour)

I built bAIkpacking Agent, a recommender system for bikepacking trips and ultra-distance events.

Problem

Bikepacking setup advice is highly fragmented. Information is spread across race reports, rider interviews, gear lists, and forums. Riders typically need to piece together this knowledge manually, which is time-consuming and inconsistent.

Solution

Baikpacking Agent centralizes this scattered knowledge into a structured system.

It accepts a natural language query and returns a context-aware recommendation, grounded in real rider data and event-specific conditions.

How it works (workflow)

Query understanding

Classifies user intent (full setup vs component)
Resolves the event (e.g., Transcontinental, GranGuanche … )
Context building

Enriches the query with event characteristics (terrain, climate, difficulty)
Retrieval planning

Decides how to fetch relevant data (exact event → similar events → global)
Knowledge retrieval

Queries a structured knowledge base (rider setups, gear mentions)
Optionally augments with web context
Recommendation generation

LLM synthesizes evidence into a structured bikepacking setup
Validation & policy layer

Ensures recommendations are grounded and avoids/ reduces hallucinations by obtaining user feedback
Tech Stack

Backend: Python, FastAPI
LLM orchestration: Pydantic-based agent design
Model: OpenAI (LLM for reasoning + generation)
Database: PostgreSQL + pgvector (embeddings & retrieval)
Data pipeline: Custom scraping, chunking, and embedding pipeline
UI: Reflex (chat interface)
Observability: Logging, traces, evaluation datasets
Evaluation: LLM-as-judge + structured test cases
Repository: https://github.com/edugonzaloalmorox/baikpacking-agent
