Day of AI Engineer

My vision of the AI Engineer role

About This Talk

  • My personal vision, based on direct experience
  • Teaching AI engineering courses, working with ML/AI for years
  • PM, DA, DS, DE, MLE, MLOps — how do these map to AI engineering?

Core Responsibility: AI Integration

  • AI Engineer = integrating AI into the product
  • In practice: calling LLM APIs (OpenAI, Anthropic, etc.)
  • Get response, integrate results into your product

Simple Example: Online Classifieds

  • Upload image, AI extracts title/description/category automatically

"That's it, right?"

Define a Pydantic class, send to OpenAI, done?

Not quite. 10 things need to happen.

The 10 Things (1/2)

  1. Prompt testing — verify the output works
  2. Iterating on prompt — change prompt, run evals, no degradation
  3. Evaluation dataset — bunch of images, measure quality
  4. Rolling out — A/B test, small portion first
  5. Production monitoring — dashboard for errors, failures

The 10 Things (2/2)

  1. Collecting logs — inspect input/output for misalignment
  2. User feedback — explicit (thumbs up/down) and implicit (corrections)
  3. Human annotators — sample and verify, add to eval set
  4. Model updates — new model from OpenAI, run evals, compare
  5. Prompt versioning — experiment tracking (MLflow/Git)

Even Simple = A Lot of Work

  • Not enough to define a Pydantic class and call OpenAI
  • UI changes needed too (frontend engineers, or AI engineers at startups)
  • CI/CD: tests on push, staging, production rollout

Complex Scenarios: RAG and Agents

  • RAG: search engine, data pipelines, infrastructure provisioning
  • Agents: tool calling, testing tools, evaluation criteria for tool usage
  • Complexity is 10x compared to the simple case

Key Skills

  1. Interacting with APIs, creating agents, RAG
  2. Testing and evaluation
  3. Production monitoring
  4. AI observability

What AI Engineers Don't Do

  • Don't create custom models from scratch
  • Don't build custom model architectures
  • Don't focus on traditional feature engineering

Instead: engineering best practices, prompt design/versioning, product integration

Part 2: Comparing Roles

AI Engineer vs ML Engineer vs Data Scientist

AI Engineer vs ML Engineer

  • ML Engineer: integrate machine learning into product
  • AI Engineer: integrate AI into product
  • Key difference: call a web service vs own the model weights
  • Sometimes AI engineers serve open-source LLMs too (vLLM, bigger GPUs)

AI Engineer vs Data Scientist

  • DS: training models, experiments, translating requirements to ML terms
  • With LLMs: the model is already there (OpenAI trained it)
  • For simple cases without fine-tuning: don't really need DS
  • Small AI team: DS role less needed

Bigger Organizations

Organizations that don't have AI engineers yet, but have MLE and Data Scientists

  • New AI use case arrives: both DS and MLE work on it
    • DS: prompt tuning, validation framework, agent behavior
    • MLE: engineering aspects, serving, CI/CD
  • They split AI engineer responsibilities between them

LLM is Not the Answer to Everything

  • Still need DS and MLE for traditional ML
  • Legacy systems, problems that don't need LLMs
  • But some things are simpler now (NLP especially)
  • Company might hire dedicated AI engineer alongside DS/MLE

The Disappearing Data Scientist

Before: model training, feature engineering, custom architectures

Now: engineering practices, evaluation, prompt versioning, A/B testing

The focus shifted from modeling to engineering.

Career Paths to AI Engineer

Data Scientist path:

  • Has: evaluation, versioning, experimental mindset
  • Needs: stronger coding, AI APIs, production practices

ML Engineer path:

  • Has: productionization, engineering practices, team collaboration
  • Needs: prompt engineering, AI evaluation patterns

Teams Without "AI" in the Name

  • Many companies: no dedicated AI engineer role
  • DS + MLE split the responsibilities
  • Job titles are old, tasks are new
  • Using OpenAI APIs instead of training models

Part 3: CRISP-DM for AI

A 1996 framework that still works

CRISP-DM

Cross-Industry Standard Process for Data Mining (1996)

Six phases: Business Understanding, Data Understanding, Data Preparation, Modeling, Evaluation, Deployment

Running Example

Same classifieds platform:

  • ML framing: classification problem
  • AI framing: upload image, extract information reliably

Business Understanding

  • Start from a real user problem, not "we have AI, let's use it"
  • What is the problem? Why solve it? Who does it affect?
  • Example: form takes 5 min, we want 1 min
  • Secondary metric: 15% abandon rate, target 5%
  • AI engineer + PM: do we need AI, or something simpler?

Data Understanding

  • Do we have the data we need to send to AI?
  • Our case: image from frontend (easy)
  • Backend AI: need data for AI to act on
  • RAG: do we have all data and integrations?
  • Some services are hard to integrate

Data Preparation

  • ML: fetch from sources, prepare format
  • Our case: image + text, fairly simple
  • For search/recommendations: need data pipelines
  • For agents: need integrations with external services

Modeling (Reframed for AI)

  • ML: train model, cross-validation
  • AI: prompt + Pydantic output class + input
  • Set up validation framework and tests
  • Collect evaluation data, define technical metrics
  • Tune prompt, iterate
  • Note: this is "modeling" in CRISP-DM, not the "evaluation" step

Evaluation

  • Roll out to users, run A/B tests
  • Check the business metric defined earlier
  • Collect implicit feedback (do users change the output?)
  • Evaluation and deployment are almost the same now
  • Possible outcome: park the project if it doesn't work

Deployment

  • Put system into production
  • Iterative: cycle through all steps until satisfactory

Who is Involved at Each Step

StepRoles
Business understandingPM, AI engineer
Data understandingDA, PM, AI engineer, backend
Data preparationDE, AI engineer
ModelingAI engineer
EvaluationAI engineer, DA, PM
DeploymentAI engineer, platform engineer

AI engineers are in every step.

The Iterative Nature

  • Never truly "finish" — iterate until satisfactory
  • User behavior unpredictable
  • Prompts need refinement
  • Evaluation metrics may need adjustment
  • Framework from 1996 still works because fundamentals don't change

Closing

  • Tools change (training models to API calls), process stays constant
  • DS processes (evaluation, versioning, experimentation) = what AI engineers need
  • DS and MLE make excellent AI engineers with additional practice
