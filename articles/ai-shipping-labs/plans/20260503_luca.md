## Plan

Gmail Triage and the two Zoomcamp capstones (circus arcade web game, modern-Python-alternatives suggester) stay parked behind Chat with My Notes for this sprint. They are good candidates for the next sprint, picked based on what feels weakest after week 6.

### Timeline

Week 1 (phase 1 starts):

- Concept and data design before any configuration. RAG quality is decided almost entirely before you start - by what is in the data, what questions you actually ask, and what shape the answer should take. Write a one-page note answering:
  - What is in the Obsidian vault? (Folder structure, note templates, frontmatter conventions, tags, embedded code or images, link graph.) Why: this decides how the vault-search tool retrieves and what makes it into the prompt. Short tag-rich notes may favour tag-filtered keyword search; long-form notes need chunking; heavy `[[wiki-link]]` use opens up small graph traversals at retrieval time. You know your vault as a user - this exercise turns that into explicit design decisions.
  - What questions do you actually want to ask it? (Pick five real questions you have asked yourself this month.) Why: real questions are your only honest evaluation on day one. Without them you configure whatever feels clever and discover at week 4 that it does not answer questions you actually have. These five questions also seed the golden eval set in week 5.
  - How would you want the answer back? (Plain text, source-linked passage, follow-up question, summary digest?) Why: a "passage with source link" is closer to search; a "summary digest" or "follow-up question" is closer to an agent. This shapes the system prompt and the tool design.
- Configure Hermes for Chat with My Notes. Define the vault-search tool (what arguments it takes, what it returns, how it retrieves), set up vault access, write a first system prompt. Phase 1 is configuration, not coding - you are giving your operational Hermes agent enough information to do the job. Why this phase first: it is the fastest path from "I have a vault and questions" to "I have a thing that answers them" - it ships early, and the shipped artefact tells you what your design decisions actually look like in practice before you commit them to code in phase 3.
- Start AI Hero in the background. Cover the agent-loop and tool-calling sections only - skip anything you already understand. Deeper study starts in week 3.
- First LinkedIn post: "Why I am building this and how I will work" - introduces the three-phase plan and Hermes as the meta-thread of the series.

Week 2 (phase 1 ships):

- Finish phase 1: get the Hermes use case answering the five questions end-to-end. Iterate on the vault-search tool (chunking, top-k, filters) and the system prompt until answers are reasonable. Done state: Chat with My Notes is a working Hermes use case you use daily.
- Begin daily use immediately. Keep a failure log - missing context, wrong passages, hallucinated facts, wrong source links. That list drives the iteration and reimplementation in weeks 3-5. Failure modes only show up under real use; a test set catches what you imagine will go wrong, daily use catches what actually does.
- Second LinkedIn post: "Chat with My Notes shipped in 2 weeks as a Hermes use case - here is what configuring it actually looked like."

Week 3 (phase 2: document and understand):

- Document how the working Hermes version actually behaves. Write a clear technical description end-to-end: the vault-search tool's contract (inputs, outputs, retrieval logic), the system prompt and why it is shaped that way, how answers come back, the failure modes from week 2's daily use. The reader is your future self in week 4 starting the reimplementation - and a hiring committee. Why this phase: jumping straight from a working Hermes use case to a framework reimplementation skips the step where you actually understand what you are about to rebuild. Documentation forces understanding; understanding makes phase 3 a focused exercise rather than a blind copy.
- Pick the agentic framework you will use for phase 3 by end of week. Skim docs and a small example for two or three candidates and choose. Selection criteria: clear modelling of tools/prompts/agent state, decent docs, works with your model provider. Pick what fits how you want to design - there is no wrong answer. Current leaning is pydantic-ai or smolagents (both lightweight, both let you see what the agent is actually doing rather than abstracting it away)[^7].
- Start deeper agents study. AI Hero on agent loops, tool calling, orchestration patterns, evaluation. The framework's docs are study material too - read them as one approach to agent design, not just API reference.
- Third LinkedIn post: "How my Hermes Chat with My Notes use case actually works - tool, prompt, retrieval, failures." This is the technical write-up; it doubles as the design spec for phase 3.

Week 4 (phase 3 starts):

- Begin phase 3: reimplement Chat with My Notes as a standalone Python application using your chosen framework. Now your code is the runtime; Hermes is no longer in the loop. Build from the design spec you wrote in week 3. Define `search_notes`, `get_note`, and `list_tags` as the framework's tools; write the system prompt; wire the agent. Keep the Hermes use case running in parallel as the reference - do not turn it off.
- By end of week, the framework version should answer the five questions end-to-end, even if rough on edges. Compare to the Hermes use case on the same questions: roughly comparable on direct lookups is the bar; gaps on multi-hop questions ("which note links to X via Y") are the iteration target for week 5.
- Continue agents study, now informed by the design choices you are making in the framework. Topics to prioritise: when to add a tool vs. extend a prompt, how to bound an agent loop, what to evaluate.
- Fourth LinkedIn post: "Day-one design decisions in [framework] - what stayed the same as the Hermes use case, what had to change, what surprised me."

Week 5 (phase 3 ships):

- Finish the framework reimplementation. On par with the Hermes use case on direct lookups; ideally better on multi-hop questions.
- Pick the single biggest failure mode from weeks 2-4 and fix it - in either implementation: better passage selection, smarter chunking, a re-ranker, a clearer system prompt, or a follow-up question loop.
- Add a small evaluation script - five to ten golden questions with expected sources or expected answer fragments. Have it grade the Hermes use case and the framework version side by side. This turns "they feel about the same" into actual numbers, which is gold for the LinkedIn write-up and the README.
- Fifth LinkedIn post: "Hermes use case vs [framework] standalone - building the same RAG agent two ways, side-by-side eval results."

Week 6 (ship):

- Wrap to a state that can be demoed at the sprint and put on the GitHub README. README explains the data structure, the working Hermes use case, and the standalone framework reimplementation - in a way a hiring committee can read.
- Tag a release. Record a short demo video (3-5 minutes) showing five real questions answered by the framework version, with the Hermes use case mentioned as the predecessor.
- Sixth LinkedIn post: "What I shipped, what is next." Link to repo, demo, and the previous five posts.
- Decide the next sprint. Likely candidates: Gmail Triage with the same three-phase pattern, or finishing one of the Zoomcamp capstones with the agent skills now in place.

### Project approach

Additional principles for taking this from idea to a shipped first version:

- First make it work, then understand it, then reimplement. Phase 1 (weeks 1-2) ships Chat with My Notes as a Hermes use case - this protects the project from joining the two unfinished capstones. Phase 2 (week 3) documents the working version so you actually understand it before rebuilding. Phase 3 (weeks 4-5) reimplements as a standalone Python application using an agentic framework of your choice. Skip phase 1 and you lose the sprint to design paralysis. Skip phase 2 and the reimplementation is blind copying. Skip phase 3 and you only learned to configure Hermes - not the skill the AI Engineer / AI systems job market is paying for.
- Phase 3 is non-negotiable. The temptation once the Hermes use case works is to call it done and move on - that leaves you with "I configured Hermes well", not "I built an agent". Week 3's documentation makes phase 3 ready to start; weeks 4-5 are structurally the reimplementation period. If phase 3 has not started by end of week 3, that is the single most important conversation to have at the week 4 check-in.
- In phase 3, tools as plain functions, callable from a REPL. The framework wires them into the agent loop, but `search_notes(query)` and similar should work standalone outside the framework. Why: when the agent misbehaves, the first question is always "is the tool wrong or is the orchestration wrong?" Plain functions let you answer that in a REPL in 30 seconds.
- One project until it ships. Chat with My Notes is the only project for this sprint. Gmail Triage and the two Zoomcamp capstones are deferred until V1 demos.
- Document weekly, not at the end. The LinkedIn posts are part of the product. Each one captures one decision while it is fresh - not a victory-lap retrospective at week 6. This matters for the job search and for the community.
- Make the tool actually useful to you. The point of "Chat with My Notes" is that you use it, not that it exists. If by week 5 you are not using it daily, fix that before adding features.
