His publication count and scientific credibility are strong; the missing piece is shipped, agent-shaped engineering that hiring committees for AI/technologist roles in research institutes can point to. One project shipped end to end, documented as he goes, closes that gap fastest.

## Plan

### Focus

- Main focus: ship a working first version of Chat with My Notes - RAG over his Obsidian vault - in 6 weeks. Build the agent loop, tools, and prompt logic by hand. Use Hermes for boilerplate only.
- Supporting focus: a public-facing LinkedIn series with Hermes as the connecting thread - one short post per week describing one technical decision and what was learned. The series is a portfolio artefact for the next job application as much as it is community sharing.
- Supporting focus: AI Hero in parallel, only the modules that fill agent-fundamentals gaps. Skim, do not complete cover-to-cover.

Gmail Triage and the two Zoomcamp capstones (circus arcade web game, modern-Python-alternatives suggester) stay parked behind Chat with My Notes for this sprint. They are good candidates for the next sprint, picked based on what feels weakest after week 6.

### Timeline

Week 1:

- Concept and data design before any code. Write a one-page note answering:
  - What is in the Obsidian vault? (Folder structure, note templates, frontmatter conventions, tags, embedded code or images, link graph.)
  - What questions do you actually want to ask it? (Pick five real questions you have asked yourself this month.)
  - How would you want the answer back? (Plain text, source-linked passage, follow-up question, summary digest?)
- Pick what gets built by hand vs delegated to Hermes. Default: agent loop, tool definitions, and prompts written by hand. Hermes is allowed for ingestion glue, Dockerfile, schema migrations, README scaffolding.
- Start AI Hero in the background. Cover the agent-loop and tool-calling sections only - skip anything you already understand.
- First LinkedIn post: "Why I am building this and how I will work" - introduces Hermes as the meta-thread of the series.

Week 2:

- Ingest the vault into a simple store - Markdown files indexed with embeddings (or even keyword search for V1). Do not over-engineer the storage - SQLite plus a small embeddings table is enough.
- Implement retrieval as a plain function: `search_notes(query) -> list[Passage]`. Make it callable from a REPL. Run the five real questions through it and check the top-k passages by hand.
- Implement the answer step as a plain function: `answer(query, passages) -> str`. Again, callable from a REPL. Compare answers to what you would write yourself.
- No agent yet - this week is about the underlying tools working correctly when called directly.

Week 3:

- Wire `search_notes` and `answer` into the smallest possible end-to-end loop. CLI is fine for V1; a single-page UI is allowed if it speeds up daily use.
- Use the tool on real questions every day this week. Keep a list of where it fails - missing context, wrong passages retrieved, hallucinated facts, wrong source links. That list drives weeks 4 and 5.
- Second LinkedIn post: "Why I built the tools as plain functions before adding an agent."

Week 4:

- Now wrap the functions in an agent. Define `search_notes`, `get_note`, and `list_tags` as tools the agent can call. Write the agent loop yourself - read other people's loops for shape but type yours from scratch.
- Compare agent answers to plain-function answers from week 3 on the same five questions. The agent should win on multi-hop questions ("which note links to X via Y") and not lose on direct lookups.
- Log every tool call to a small file or table - decisions, latency, token cost. This is the foundation for the monitoring step later.

Week 5:

- Iterate based on what week 3-4 surfaced. Pick the single biggest failure mode and fix it: better passage selection, smarter chunking, a re-ranker, a clearer system prompt, or a follow-up question loop.
- Add a small evaluation script - five to ten golden questions with expected sources or expected answer fragments. This is your week-by-week regression check from now on.
- Third LinkedIn post: "Building an agent from scratch vs. using one - what I learned about Hermes by writing my own."

Week 6:

- Wrap to a state that can be demoed at the sprint and put on the GitHub README. README explains the data structure, the manual workflow, and what the agent automates - in a way a hiring committee can read.
- Tag a release. Record a short demo video (3-5 minutes) showing five real questions answered by the system.
- Fourth LinkedIn post: "What I shipped, what is next." Link to repo, demo, and the previous three posts.
- Decide the next sprint. Likely candidates: Gmail Triage with the same build pattern, or finishing one of the Zoomcamp capstones with the agent skills now in place.

### Project approach

Additional principles for taking this from idea to a shipped first version:

- Build the agent yourself. The whole point is to learn how an agent loop, tool definitions, and prompts hang together. Hermes is great for productivity; it is not the right tool for the part you are trying to learn. Type the agent loop by hand, even if it is short.
- Tools as plain functions first, then agent. If `search_notes(query)` and `answer(query, passages)` work when called directly from a REPL, the agent's job becomes orchestration rather than first-pass logic. Debugging is far easier.
- One project until it ships. Chat with My Notes is the only project for this sprint. Gmail Triage and the two Zoomcamp capstones are deferred until V1 demos.
- Document weekly, not at the end. The LinkedIn posts are part of the product. Each one captures one decision while it is fresh - not a victory-lap retrospective at week 6. This matters for the job search and for the community.
- Make the tool actually useful to you. The point of "Chat with My Notes" is that you use it, not that it exists. If by week 5 you are not using it daily, fix that before adding features.

### Job positioning

Luca specifically asked for guidance on framing his profile toward AI Engineer / technologist / AI systems roles in research institutes without losing credibility in academic selection. Concrete pointers:

- Lead with the combination, not the transition. "Research scientist who ships production-grade AI tooling" reads stronger than "researcher transitioning to AI engineering". Publications stay where they belong (CV, Google Scholar); the shipped agent goes next to them, not instead of them.
- One shipped, well-documented agent project beats a long list of half-built ones. Chat with My Notes plus a public LinkedIn write-up is the single strongest signal for AI Engineer / AI systems roles - especially in research institutes, where the audience values both rigor and pragmatism.
- The GAIL certification is a useful credential but not a substitute for evidence. Mention it; do not lead with it.
- Use the AI Engineering Field Guide ([github.com/alexeygrigorev/ai-engineering-field-guide](https://github.com/alexeygrigorev/ai-engineering-field-guide)) to scan recent AI Engineer / AI systems job listings - especially in research-adjacent or scientific institutions. The point is to harvest the actual responsibilities and skills they list, then mirror that language in your CV and LinkedIn headline.
- For researcher / technologist roles in research institutes, the LinkedIn series is doubly useful: it shows you can write up technical work, which is the bridge between a publication and an engineering blog post.
